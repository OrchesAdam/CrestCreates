# CrestCreates 跨项目模块发现系统

## 概述

CrestCreates 的跨项目模块发现系统是一个基于 MSBuild 的编译时模块扫描和注册机制。它允许框架自动发现分布在多个项目中的模块，生成类型安全的注册代码，并按照依赖关系进行拓扑排序。

## 核心特性

- **编译时扫描**: 在构建过程中自动扫描所有项目中的模块
- **跨项目支持**: 自动发现引用项目中的模块
- **类型安全**: 生成强类型的模块注册代码，无需运行时反射
- **拓扑排序**: 根据模块依赖关系和 Order 值自动排序
- **钻石依赖处理**: 正确处理复杂的依赖关系图

## 架构组件

### 1. BuildTasks 项目

位于 `build/CrestCreates.BuildTasks/`，包含三个核心 MSBuild 任务：

#### ScanModulesFromSource
- **功能**: 扫描源代码中的 `[Module]` 属性
- **输入**: 项目源文件列表
- **输出**: `ModuleManifest.json` - 包含模块元数据
- **扫描内容**:
  - 模块类名和完整类型名
  - 依赖模块（`typeof(DependencyModule)`）
  - Order 值
  - AutoRegisterServices 设置

#### CollectModuleManifests
- **功能**: 收集当前项目和所有引用项目的模块清单
- **输入**: 
  - 当前项目的 `ModuleManifest.json`
  - 引用项目的模块清单路径
- **输出**: `AggregatedModuleManifest.json` - 聚合所有模块

#### GenerateAggregatedModuleCode
- **功能**: 生成类型安全的模块注册和初始化代码
- **输入**: `AggregatedModuleManifest.json`
- **输出**: `ModuleAutoInitializer.g.cs` - 生成的 C# 代码
- **生成内容**:
  - 模块的 DI 注册
  - 拓扑排序后的初始化顺序
  - 生命周期方法调用

### 2. MSBuild Targets 文件

`CrestCreates.Modules.props` 定义了构建流程：

```xml
<Target Name="GenerateModuleManifest" BeforeTargets="BeforeBuild">
  <!-- 扫描当前项目的模块 -->
</Target>

<Target Name="CollectAndGenerateAggregatedModules" AfterTargets="GenerateModuleManifest">
  <!-- 收集所有模块并生成代码 -->
</Target>
```

### 3. 模块清单格式

```json
[
  {
    "FullName": "CrestCreates.Core.CoreModule",
    "Name": "CoreModule",
    "Namespace": "CrestCreates.Core",
    "DependsOn": [],
    "Order": -200,
    "AutoRegisterServices": true
  }
]
```

## 使用指南

### 1. 在项目中启用模块发现

在 `.csproj` 文件中导入 props 文件：

```xml
<Project Sdk="Microsoft.NET.Sdk">
  <Import Project="..\..\..\build\CrestCreates.BuildTasks\CrestCreates.Modules.props" />
  
  <!-- 其他配置 -->
</Project>
```

### 2. 定义模块

创建实现 `IModule` 接口的类，并添加 `[Module]` 属性：

```csharp
using CrestCreates.Modularity;
using CrestCreates.Domain.Shared.Attributes;

namespace MyNamespace;

[Module(typeof(DependencyModule1), typeof(DependencyModule2), Order = 0)]
public class MyModule : IModule
{
    public string Name => "MyModule";
    public string? Description => "模块描述";
    public string? Version => "1.0.0";

    public void OnPreInitialize() { }
    public void OnInitialize() { }
    public void OnPostInitialize() { }
    public void OnConfigureServices(IServiceCollection services) { }
    public void OnApplicationInitialization(IHost host) { }
}
```

### 3. 模块属性参数

- **typeof(DependencyModule)**: 指定依赖的模块，可多个
- **Order**: 模块初始化顺序，值越小越早初始化
- **AutoRegisterServices**: 是否自动注册服务（默认 true）

### 4. 在应用程序中使用

生成的代码会自动创建 `ModuleAutoInitializer` 类：

```csharp
// 在 Program.cs 或 Startup.cs 中
var builder = Host.CreateDefaultBuilder(args);

// 注册所有模块
builder = ModuleAutoInitializer.RegisterModules(builder);

var host = builder.Build();

// 初始化所有模块
ModuleAutoInitializer.InitializeAllModules(host.Services);

host.Run();
```

## 拓扑排序算法

系统使用深度优先搜索（DFS）算法进行拓扑排序：

1. **排序依据**:
   - 首先按 `Order` 值排序
   - 然后按依赖关系排序（依赖项先于被依赖项）

2. **算法步骤**:
   ```
   1. 按 Order 值对模块进行初始排序
   2. 对每个未访问的模块执行 DFS
   3. 在 DFS 中，先递归访问所有依赖项
   4. 将当前模块添加到已排序列表
   5. 检测循环依赖（如果在访问中遇到已访问节点）
   ```

3. **示例排序结果**:
   ```
   CoreModule (Order = -200)           ← 最先初始化
   CoreModule (Order = -100)           
   DataModule (Order = -100, 依赖 CoreModule)
   CachingModule (Order = -50, 依赖 CoreModule)
   DatabaseModule (Order = -50, 依赖 CoreModule)
   ApplicationModule (Order = 0, 依赖 DatabaseModule, CachingModule, DataModule)
   WebModule (Order = 100, 依赖 ApplicationModule)  ← 最后初始化
   ```

## 跨项目模块发现

### 工作原理

1. **构建顺序**:
   - MSBuild 按照项目依赖顺序构建
   - 引用项目先于当前项目构建
   - 每个项目生成自己的 `ModuleManifest.json`

2. **模块收集**:
   - `CollectModuleManifests` 任务接收引用项目路径
   - 在每个引用项目的 `obj/$(Configuration)/$(TargetFramework)/` 目录中查找清单
   - 聚合所有模块到一个清单文件

3. **代码生成**:
   - 基于聚合后的清单生成统一的注册代码
   - 包含所有项目中的模块

### 项目结构示例

```
CrestCreates.Web/                    ← Web 项目（入口点）
├── obj/Debug/net10.0/
│   ├── ModuleManifest.json          ← 5 个 Web 项目模块
│   ├── AggregatedModuleManifest.json ← 7 个模块（包含引用项目）
│   └── ModuleAutoInitializer.g.cs   ← 生成的注册代码
└── 引用项目:
    ├── CrestCreates.Core/           ← Core 模块项目
    │   └── obj/Debug/net10.0/
    │       └── ModuleManifest.json  ← 1 个 Core 模块
    └── CrestCreates.Data/           ← Data 模块项目
        └── obj/Debug/net10.0/
            └── ModuleManifest.json  ← 1 个 Data 模块
```

## 钻石依赖处理

系统正确处理钻石依赖场景：

```
       CoreModule (-200)
        /           \
   DataModule     CachingModule
   (-100)          (-50)
        \           /
       ApplicationModule (0)
```

**处理结果**:
- CoreModule 只初始化一次
- DataModule 和 CachingModule 按 Order 排序
- ApplicationModule 在所有依赖项之后初始化

## 生成的代码结构

### ModuleAutoInitializer.g.cs

```csharp
// <auto-generated/>
namespace CrestCreates.Web;

public static partial class ModuleAutoInitializer
{
    private static readonly List<ModuleDescriptor> _registeredModules = new();

    // 注册所有模块到 DI 容器
    public static IServiceCollection RegisterAllModules(IServiceCollection services)
    {
        // 按拓扑顺序注册
        services.AddSingleton<CrestCreates.Core.CoreModule>();
        services.AddSingleton<CrestCreates.Web.Modules.CoreModule>();
        // ... 其他模块
        
        // 记录模块描述符
        _registeredModules.Add(new ModuleDescriptor { ... });
        
        return services;
    }

    // 初始化所有模块
    public static void InitializeAllModules(IServiceProvider serviceProvider)
    {
        // 按拓扑顺序调用生命周期方法
        var coreModule = serviceProvider.GetService<CrestCreates.Core.CoreModule>();
        coreModule?.OnPreInitialize();
        // ... 其他模块
        
        coreModule?.OnInitialize();
        // ...
        
        coreModule?.OnPostInitialize();
        // ...
    }
}
```

## 配置选项

### 禁用模块生成

在 `.csproj` 中设置：

```xml
<PropertyGroup>
  <EnableModuleManifestGeneration>false</EnableModuleManifestGeneration>
</PropertyGroup>
```

### 自定义输出路径

```xml
<PropertyGroup>
  <ModuleManifestOutputPath>$(MSBuildProjectDirectory)\CustomPath\ModuleManifest.json</ModuleManifestOutputPath>
  <AggregatedModuleCodePath>$(MSBuildProjectDirectory)\Generated\ModuleAutoInitializer.g.cs</AggregatedModuleCodePath>
</PropertyGroup>
```

## 故障排除

### 常见问题

1. **模块未被发现**
   - 检查 `[Module]` 属性是否正确添加
   - 确认类实现了 `IModule` 接口
   - 验证 props 文件是否正确导入

2. **依赖关系错误**
   - 检查 `typeof(DependencyModule)` 是否正确
   - 确认依赖模块的命名空间正确
   - 查看生成的清单文件验证依赖

3. **编译错误**
   - 确保所有项目都引用了 `CrestCreates.Modularity`
   - 检查 `IModule` 接口的所有方法是否实现
   - 验证模块类的访问修饰符为 public

### 调试技巧

1. **查看生成的清单**:
   ```bash

   cat obj/Debug/net10.0/AggregatedModuleManifest.json
   ```

2. **查看生成的代码**:

## 最佳实践

1. **模块命名**: 使用有意义的模块名称，如 `CoreModule`, `DatabaseModule`
2. **Order 值**: 使用负数表示基础模块，正数表示高级模块
3. **依赖声明**: 明确声明所有依赖，不要依赖隐式顺序
4. **单一职责**: 每个模块负责一个明确的功能领域
5. **文档注释**: 为模块添加 XML 文档注释，说明其用途

```csharp
private string GenerateCode(List<ModuleManifest> modules)

    // 自定义代码生成逻辑
```
