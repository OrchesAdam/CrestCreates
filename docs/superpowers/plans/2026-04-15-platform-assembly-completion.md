### ModuleAutoInitializer Issue

**Problem:** There are TWO ModuleAutoInitializer implementations:
1. `samples/LibraryManagement/LibraryManagement.Web/Modules/ModuleAutoInitializer.cs` (manual, in source)
2. `obj/Debug/net10.0/ModuleAutoInitializer.g.cs` (generated by build task)

The generated code has `namespace LibraryManagement.Web;` while the manual one has `namespace LibraryManagement.Web.Modules;`.

**Build System Issue:** The generated `ModuleAutoInitializer.g.cs` is NOT being compiled by MSBuild despite being added to Compile items in the target. This is a timing/ordering issue in the build system.

**Current Solution:** The Sample keeps the manual ModuleAutoInitializer since it works. The generated code in obj is not used.

### Framework Startup Issue

The framework `Startup.cs` manually registers ~50+ services instead of using module auto-initialization. This is by design for the framework Web project, but creates inconsistency.

## Task 1: Define Formal Assembly Entry Points (DONE)

**Status:** Documented in this plan.

### Rules for Assembly Entry Points

1. **Framework Core Services** (always present):
   - Modularity system itself
   - Multi-tenancy
   - Basic DI container

2. **Module-Autoinitializer Services** (auto-registered via modules):
   - All modules decorated with `[CrestModule]` get `AddXxxModule()` extension
   - Module lifecycle: OnConfigureServices → OnPreInitialize → OnInitialize → OnPostInitialize
   - Services registered in `OnConfigureServices`
   - Middleware registered in `OnApplicationInitialization`

3. **Host-Required Services** (explicit opt-in):
   - Database connection strings (environment-specific)
   - Authentication providers (OAuth, etc.)
   - Third-party cloud services
