# CrestCreates 核心能力收口任务清单

## 文档目的

本文档用于将当前 `CrestCreates` 与 `ABP Framework` 的核心能力差距，整理为可执行的任务清单。

约束原则：

- 只围绕框架主链收口，不做兼容性方案
- 不引入双轨长期并存
- 不做补丁式扩展
- 每项任务都必须以平台能力闭环为目标
- 每项任务都必须可被真实验收，而不是只看代码是否存在

---

## 当前判断摘要

当前项目已经具备以下核心骨架：

- DDD、Repository、UnitOfWork、多 ORM 抽象
- Dynamic API 编译期生成主链
- 多租户主链与 Tenant Management
- Permission / RBAC 主链
- Setting Management
- Feature Management
- Audit Logging
- Event Bus / Distributed Event 基础能力
- Caching 基础能力

当前最值得优先收口的缺口，不是“再堆新模块”，而是以下三类平台能力：

1. 认证与身份主链统一
2. 后台作业平台化
3. 文件存储与模块资源能力平台化

---

## 优先级总览

| 优先级 | 任务名称 | 目标 |
|---|---|---|
| P0 | 统一认证与身份主链 | 建立唯一认证真相来源，完成身份、令牌、权限、租户上下文的统一主链 |
| P1 | 后台作业平台化 | 将现有调度能力收口为正式后台作业主链 |
| P1 | 文件存储平台化 | 将 FileManagement 收口为正式 Blob/File Storage 能力 |
| P1 | 虚拟文件系统平台化 | 建立模块资源、嵌入资源、静态资源的统一访问主链 |
| P2 | 本地化主链收口 | 将现有 Localization 从“可用”提升为“平台闭环” |
| P2 | 平台装配入口收口 | 将 Web / Sample 中的能力装配方式收口为一致主链 |
| P2 | 核心能力主链集成验收 | 建立面向主链的跨模块真实集成测试资产 |

---

## P0：统一认证与身份主链(已完成)

### 需要实现什么

建立唯一正式认证主链，明确：

- 用户、角色、权限的唯一身份主真相
- Token 签发与校验的唯一主链
- 租户上下文如何进入认证票据与权限校验链
- OpenIddict、JwtBearer、应用身份服务之间的职责边界

同时清理当前分散状态，禁止继续演化为多套认证真相来源并存。

### 拆分任务

#### 任务 1：定义认证主链边界(已完成)

完成哪些功能：

- 明确 `AuthService`、`OpenIddict`、`JwtBearer`、`IdentityManagement` 的职责边界
- 明确谁负责登录、签发 Token、刷新 Token、用户信息解析、权限装载
- 明确租户上下文进入 claims / token / permission 的正式链路
- 明确哪些旧入口不是正式主链

预期结果：

- 输出唯一认证主链设计
- 项目中不存在语义重叠且长期并存的认证入口

验收清单：

- 能明确指出登录入口、Token 签发入口、Token 校验入口各自唯一实现
- 能明确指出权限来源与租户来源的唯一读取路径
- 能明确说明哪些旧链路不再作为正式主链维护

#### 任务 2：收口 Token 签发与验证链路(已完成)

完成哪些功能：

- 统一 OpenIddict 与 JwtBearer 的使用方式
- 保证访问令牌、刷新令牌、用户信息接口链路一致
- 保证 Token 中的身份、角色、租户、权限相关信息来源一致

预期结果：

- 认证服务可以稳定完成登录、刷新、鉴权、获取当前用户
- 租户上下文与权限校验能够在认证后自然贯通

验收清单：

- 登录成功后可获得有效访问令牌
- 刷新令牌流程可正常工作
- 受保护接口可通过统一认证主链访问
- 当前用户、当前租户、权限校验结果在请求链路中一致

#### 任务 3：身份管理与认证链路整合(已完成)

完成哪些功能：

- `UserAppService`、`RoleAppService` 与认证主链整合
- 用户启用/禁用、密码修改、角色变更后，认证行为和权限行为一致
- 统一安全日志写入点

预期结果：

- Identity Management 不再只是 CRUD，而是认证主链的一部分

验收清单：

- 新建用户后可按预期登录
- 修改密码后旧密码失效、新密码生效
- 角色调整后权限结果符合预期
- 关键认证行为可写入统一安全日志

#### 任务 4：移除非正式认证路径的扩张空间(已完成)

完成哪些功能：

- 禁止继续在空壳或旁路模块中增加认证真相来源
- 对空壳认证模块进行清理或明确降级状态
- 避免 OAuth / Jwt / OpenIddict 各自长出独立业务逻辑

预期结果：

- 后续认证演进只能走唯一主链

验收清单：

- 仓库中不存在“另起一套用户认证逻辑”的新入口
- 空壳模块不再误导维护者

## P1：后台作业平台化

### 需要实现什么

将现有 `Scheduling` / `Quartz` 能力从“调度器封装”提升为“后台作业平台能力”，形成统一作业定义、注册、调度、执行、失败处理与观测主链。

### 拆分任务

#### 任务 1：定义后台作业抽象

完成哪些功能：

- 定义统一后台作业模型，而不是只暴露 Quartz API
- 明确一次性作业、延迟作业、周期作业的统一抽象
- 明确作业参数传递、执行上下文、租户上下文传递规则

预期结果：

- 上层模块面向平台作业抽象，而不是直接依赖 Quartz 细节

验收清单：

- 业务模块不需要直接引用 Quartz 即可定义作业
- 作业执行时可拿到明确的上下文与参数

#### 任务 2：统一作业注册与执行主链

完成哪些功能：

- 建立统一作业注册入口
- 建立统一作业调度入口
- 建立统一执行包装：日志、异常、租户上下文、权限边界

预期结果：

- 框架可稳定注册、调度、执行后台作业

验收清单：

- 可注册一个后台作业并成功执行
- 可调度立即执行、延迟执行、Cron 执行
- 作业执行失败时可记录错误并可追踪

#### 任务 3：补齐作业持久化与失败处理

完成哪些功能：

- 定义失败处理策略
- 定义重试策略
- 定义作业状态记录方式

预期结果：

- 后台作业具备最小平台可靠性，而不是仅能“触发”

验收清单：

- 执行失败后可看到状态变化
- 可按配置进行重试
- 重试后状态正确收敛

#### 任务 4：建立后台作业真实集成测试

完成哪些功能：

- 针对主链编写真实集成测试
- 验证租户上下文、日志、异常、调度行为

预期结果：

- 后台作业能力被正式纳入主测试资产

验收清单：

- 至少覆盖一次性、延迟、周期三类作业
- 至少覆盖一次失败与重试场景
- 至少覆盖一个带租户上下文的作业场景

## P1：文件存储平台化

### 需要实现什么

将现有 `FileManagement` 从“本地文件服务”收口为正式 Blob/File Storage 平台能力，明确统一文件抽象、存储提供者模型、元数据、访问地址与多租户边界。

### 拆分任务

#### 任务 1：定义文件存储主模型

完成哪些功能：

- 定义统一文件对象模型
- 定义文件键、租户边界、内容类型、大小、扩展名、存储提供者语义
- 定义上传、下载、删除、替换、访问地址生成规则

预期结果：

- 文件存储成为平台抽象，而不是本地文件系统包装

验收清单：

- 上层调用不依赖本地磁盘路径语义
- 文件模型能够表达租户隔离和提供者差异

#### 任务 2：收口当前本地文件实现

完成哪些功能：

- 将当前 LocalFileSystemProvider 纳入统一主链
- 规范目录结构、命名规则、访问路径生成
- 补齐异常处理和输入约束

预期结果：

- 本地文件存储成为正式 provider，而不是临时实现

验收清单：

- 可稳定上传、读取、删除文件
- 文件路径规则一致
- 多租户场景下不会串租户

#### 任务 3：设计并接入第二存储提供者

完成哪些功能：

- 基于已有抽象接入一个正式云存储 provider
- 验证 provider 模型不是为本地文件特化

预期结果：

- 文件存储抽象经受跨 provider 验证

验收清单：

- 切换 provider 不影响上层调用方式
- 元数据、访问地址、租户边界行为一致

#### 任务 4：建立文件存储真实测试资产

完成哪些功能：

- 为文件上传、读取、删除、替换、租户隔离建立测试

预期结果：

- 文件存储成为可被稳定回归验证的平台能力

验收清单：

- 至少覆盖本地 provider
- 至少覆盖一个跨租户隔离测试
- 至少覆盖异常输入与非法访问测试

---

## P1：虚拟文件系统平台化

### 需要实现什么

建立模块资源、嵌入资源、静态资源的统一访问主链，避免文档、模板、模块资源继续散落在各模块各自处理。

### 拆分任务

#### 任务 1：定义虚拟文件系统抽象

完成哪些功能：

- 定义虚拟路径模型
- 定义文件集、模块资源、嵌入资源统一抽象
- 定义资源读取与列举接口

预期结果：

- 模块资源访问不再直接依赖物理路径

验收清单：

- 能通过统一接口读取模块资源
- 调用方不需要关心资源来自磁盘还是嵌入资源

#### 任务 2：接入模块资源主链

完成哪些功能：

- 将框架文档模板、代码生成模板、模块静态资源纳入统一资源访问链

预期结果：

- 模块可声明和暴露自己的资源文件集

验收清单：

- 代码生成模板可通过统一 VFS 访问
- 模块资源可被发现和读取

#### 任务 3：建立资源访问测试

完成哪些功能：

- 覆盖资源发现、读取、冲突处理、模块隔离

预期结果：

- VFS 成为平台级基础设施，而不是文档概念

验收清单：

- 至少覆盖磁盘资源与嵌入资源两类来源
- 至少覆盖模块资源冲突或命名空间隔离

---

## P2：本地化主链收口

### 需要实现什么

将当前本地化能力从 provider 级实现，提升为完整平台能力，包括资源组织、文化解析、请求链路接入与模块协作方式。

### 拆分任务

#### 任务 1：定义本地化资源主链

完成哪些功能：

- 明确资源定义方式
- 明确模块如何贡献资源
- 明确多文化回退规则

预期结果：

- 本地化资源有统一组织方式

验收清单：

- 新模块能按统一约定提供本地化资源
- 文化回退行为可预测

#### 任务 2：补齐请求文化接入主链

完成哪些功能：

- 将请求文化解析纳入 Web 主链
- 明确 Header、QueryString、Cookie 等文化选择规则

预期结果：

- 本地化不再只是服务内部调用能力

验收清单：

- 请求可切换文化
- 相同资源在不同文化下返回正确内容

#### 任务 3：建立模块化本地化测试

完成哪些功能：

- 覆盖文化切换、资源回退、模块资源加载

预期结果：

- 本地化主链可被真实回归验证

验收清单：

- 至少覆盖两种文化
- 至少覆盖缺失资源回退

---

## P2：平台装配入口收口

### 需要实现什么

将当前 `Startup`、`Sample Program.cs`、模块初始化、各类 `AddXxx` 装配方式收口为一致入口，减少样板式重复拼装和隐式遗漏。

### 拆分任务

#### 任务 1：梳理正式装配入口

完成哪些功能：

- 明确哪些服务属于框架默认主链
- 明确哪些能力应通过模块自动初始化装配
- 明确哪些能力只能由宿主显式选择接入

预期结果：

- 宿主装配方式有统一规则

验收清单：

- 示例项目与框架默认宿主的装配方式一致
- 新增模块不会要求宿主反复手工拼接低层依赖

#### 任务 2：收口示例项目的装配方式

完成哪些功能：

- 清理 Sample 中与框架主链不一致的装配代码
- 让 Sample 成为正式主链示例，而不是“还能跑的另一套写法”

预期结果：

- Sample 成为主链使用说明，而不是旁路实现

验收清单：

- Sample 中不再维护明显偏离主链的初始化方式
- Sample 能体现模块自动初始化、Dynamic API 生成链、设置/特性/租户主链

## P2：核心能力主链集成验收

### 需要实现什么

建立面向正式主链的真实集成测试资产，避免“模块都有，但平台闭环未验证”的状态长期存在。

### 拆分任务

#### 任务 1：建立核心验收矩阵

完成哪些功能：

- 明确哪些能力必须通过真实集成测试验收
- 明确验收链路覆盖范围

预期结果：

- 主测试资产围绕正式主链组织

验收清单：

- 至少覆盖认证、租户、权限、Setting、Feature、Dynamic API、Audit Log

#### 任务 2：补齐跨模块联动场景

完成哪些功能：

- 验证认证 + 租户 + 权限
- 验证租户 + Setting + Feature
- 验证 Dynamic API + 权限 + 审计
- 验证后台作业 + 租户 + 日志
- 验证文件存储 + 租户 + 权限

预期结果：

- 不再只验证单模块可用，而是验证平台闭环成立

验收清单：

- 每条关键业务链路都有对应真实测试
- 测试表达的是正式主链，而不是 legacy path

---

## 交付顺序建议

建议严格按以下顺序推进：

1. P0 统一认证与身份主链
2. P1 后台作业平台化
3. P1 文件存储平台化
4. P1 虚拟文件系统平台化
5. P2 本地化主链收口
6. P2 平台装配入口收口
7. P2 核心能力主链集成验收

原因：

- 认证主链决定权限、租户、身份的统一真相来源
- 后台作业和文件能力是典型平台能力，必须尽早收口
- VFS 和本地化属于模块化平台能力的基础设施
- 装配入口和集成验收应在主链边界稳定后再统一收口

---

## 完成定义

每项任务完成时，必须同时满足以下条件：

- 主链唯一且清晰
- 不新增 runtime fallback 或双轨正式实现
- 有真实集成测试或等价的全链路验收
- 宿主接入方式清晰且不依赖隐式约定
- 不误导后续维护者继续维护旁路实现

如果只完成了代码存在、接口存在、示例存在，但没有主链闭环和真实验收，则该任务不视为完成。
